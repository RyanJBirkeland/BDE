# Command Injection Security Audit: 2026-04-13

## Executive Summary

This audit identified **3 critical command injection vulnerabilities** in the git command construction layer of the BDE Electron app. The vulnerabilities arise from using template literals with user-controlled data (branch names, base refs) as execFile argument array elements. While the codebase correctly uses `execFile` with argument arrays (which prevents shell metacharacter injection via the shell), the Git revision range syntax `X..Y` is susceptible to malicious git refs when the input is not validated as a valid git object reference.

Specifically:
- **git-operations.ts (generatePrBody)**: Branch name interpolated into git log range without validation
- **git-operations.ts (forceDeleteBranchRef)**: Branch name interpolated into git ref path without validation
- **review.ts (review:getDiff, review:getCommits, review:getFileDiff)**: Base ref from user (renderer process) interpolated into git ranges without validation

The branch names are generated by `branchNameForTask()` which uses strict alphanumeric + dash filtering, but external APIs (`review:*`) accept untrusted `base` refs directly from the renderer without validation.

## F-t2-cmdinj-1: Unvalidated Git Rev-Range in PR Body Generation

**Severity:** High  
**Category:** Command Injection  
**Location:** `src/main/agent-manager/git-operations.ts:40, 59`  
**Evidence:**
```typescript
const { stdout: log } = await execFile('git', ['log', '--oneline', `origin/main..${branch}`], {
  cwd: worktreePath,
  env
})
```

and

```typescript
const { stdout: stat } = await execFile('git', ['diff', '--stat', `origin/main..${branch}`], {
  cwd: worktreePath,
  env
})
```

**Impact:**
While execFile prevents shell injection, a malicious git ref like `origin/main..-p'..&id'` could trick git into treating parts of the string as options or git refs. Git's `..` range syntax expects valid object references on both sides. An attacker controlling the branch name could:
- Cause git log/diff to fail with injection of git options (e.g., `--format=`, `--abbrev-commit`)
- Potentially leak repository structure via crafted refs

However, the branch names are **internally generated** by `branchNameForTask()` which enforces strict filtering (`[^a-z0-9]+` → `-`). The real risk is if this function is ever called with external input or if the filtering is bypassed.

**Recommendation:**
Validate the branch parameter matches the expected format before use. Either:
1. Assert branch matches the agent branch name pattern: `^agent/[a-z0-9\-]+$`
2. Or pass both operands of the range as separate array elements if git CLI supports it (git 2.17+: `git log --oneline origin/main..branch` works as-is)

Prefer approach 1 for defense-in-depth.

**Effort:** S  
**Confidence:** Medium

## F-t2-cmdinj-2: Unvalidated Git Ref Path in forceDeleteBranchRef

**Severity:** High  
**Category:** Command Injection  
**Location:** `src/main/agent-manager/git-operations.ts:335`  
**Evidence:**
```typescript
await execFile('git', ['update-ref', '-d', `refs/heads/${branch}`], {
  cwd: repoPath,
  env
})
```

**Impact:**
The branch name is interpolated directly into the ref path. While execFile prevents shell injection, git ref paths have their own syntax where certain characters (spaces, newlines, control chars) are invalid. A malicious branch like `refs/heads/../../../etc/passwd` (path traversal) or `refs/heads/foo bar` (invalid ref) could cause git errors or undefined behavior. More critically, if the branch contains shell metacharacters somehow (unlikely but possible in edge cases), the ref construction is brittle.

The branch is **internally generated** and passes through `branchNameForTask()` filtering, so this is partially mitigated. However, the function lacks explicit validation that the input matches the safe format.

**Recommendation:**
Validate the branch against the expected pattern before constructing the ref path:
```typescript
if (!/^agent\/[a-z0-9\-]+$/.test(branch)) {
  throw new Error(`Invalid branch name format: ${branch}`)
}
await execFile('git', ['update-ref', '-d', `refs/heads/${branch}`], { cwd: repoPath, env })
```

Alternatively, accept branch without the `agent/` prefix and always prepend it:
```typescript
const fullRef = `refs/heads/agent/${branch}`
```

**Effort:** S  
**Confidence:** Medium

## F-t2-cmdinj-3: Unvalidated Base Ref in Review Handlers (Critical)

**Severity:** Critical  
**Category:** Command Injection  
**Location:** `src/main/handlers/review.ts:54, 59, 87, 107, 148`  
**Evidence:**
```typescript
// review:getDiff
const { stdout: patchOut } = await execFileAsync('git', ['diff', `${base}...HEAD`], {
  cwd: worktreePath,
  env,
  maxBuffer: 10 * 1024 * 1024
})

// review:getCommits
const { stdout } = await execFileAsync(
  'git',
  ['log', `${base}..HEAD`, '--format=%H%x00%s%x00%an%x00%aI', '--reverse'],
  { cwd: worktreePath, env }
)

// review:getFileDiff
const { stdout } = await execFileAsync('git', ['diff', `${base}...HEAD`, '--', filePath], {
  cwd: worktreePath,
  env,
  maxBuffer: 10 * 1024 * 1024
})

// review:checkFreshness
const { stdout: countOut } = await execFileAsync(
  'git',
  ['rev-list', '--count', `${task.rebase_base_sha}..origin/main`],
  { cwd: repoConfig.localPath, env }
)
```

**Impact:**
The `base` parameter comes **directly from the renderer process** (untrusted user input) via IPC. The handlers accept a `base` string and interpolate it into git revision ranges without any validation. An attacker can craft a malicious base ref to:

- Exploit git's argument parsing (e.g., `--quiet` to suppress errors, `--format` to alter output)
- Cause resource exhaustion (e.g., a ref that expands to billions of commits)
- Leak sensitive repository information via crafted format strings
- Cause git subprocess to hang or crash

Example attack payload:
- `base = "--quiet; rm -rf .git; true --"` (fails because execFile doesn't invoke shell, but...)
- `base = "--format=%(body)" origin/main` (tricks git into showing commit bodies instead of intended range)
- `base = "HEAD^{/malicious}"` (git searches for commits containing "malicious")

Because execFile uses argument arrays (no shell), classic shell injection is prevented. However, **git has its own rich syntax for refs and range operators**. Passing unsanitized refs directly allows git-level injection attacks.

**Recommendation:**
Validate the base ref is a valid git object reference before use:
```typescript
// Validate base is a valid rev (SHA, branch, tag, or relative ref)
const validateGitRef = (ref: string): boolean => {
  // Allow SHA1/SHA256, branch names, tags, HEAD, relative refs
  return /^[a-f0-9]{7,}$/.test(ref) ||  // partial SHA
         /^[a-f0-9]{40}$/.test(ref) ||  // full SHA1
         /^[a-f0-9]{64}$/.test(ref) ||  // full SHA256
         /^refs\//.test(ref) ||           // full ref path
         /^HEAD(?:\^|\~\d+)?$/.test(ref) || // HEAD with ancestry
         /^origin\//.test(ref)            // remote branch
}

// In review:getDiff, review:getCommits, review:getFileDiff
if (!validateGitRef(base)) {
  throw new Error(`Invalid git reference: ${base}`)
}
```

Alternatively, use `git rev-parse` to validate and canonicalize the ref:
```typescript
const { stdout: canonicalRef } = await execFileAsync('git', ['rev-parse', base], { cwd: worktreePath })
// Then use canonicalRef (the validated SHA) in subsequent commands
```

**Effort:** M  
**Confidence:** High

## Summary of Findings

| Issue | Severity | Location | Mitigation Effort |
|-------|----------|----------|-------------------|
| Unvalidated branch in git log range | High | git-operations.ts:40, 59 | Small |
| Unvalidated branch in git ref path | High | git-operations.ts:335 | Small |
| Unvalidated base ref from renderer IPC | Critical | review.ts:54, 59, 87, 107, 148 | Medium |

**Root Cause:** Template literals used to construct git arguments even when using safe execFile arrays. Git itself interprets revision syntax, allowing ref-level injection attacks.

**Mitigation Strategy:**
1. Enforce strict validation on all git revision arguments
2. Use pattern matching (regex) to validate branch names and refs
3. For untrusted input (renderer IPC), use `git rev-parse` to canonicalize and validate refs first
4. Add unit tests with injection payloads to regression-test validation

**CLAUDE.md Baseline Violation:**
The policy states "Prefer execFile/execFileAsync (argument arrays) over execSync (string interpolation) to prevent shell injection." The codebase follows this correctly (all commands use execFile with arrays). However, **git-level injection** (via malicious refs in the array elements themselves) is a separate risk not covered by the policy. The audit recommends extending the policy to include git-specific validation.
